    • 可用HTML建立web表單，讓使用者輸入資訊，接著讓網頁瀏覽器將表單傳給伺服器

Request and Response

Application and Request Context

  • 當Flask從用戶端收到request時，必須讓處理它的view函式能夠使用一些物件

    • 其中例子：request物件，其封裝用戶端送來的HTTP request

  • 為了讓view函式讀取request物件，其方法有兩種：

    1. 最直接做法是用引數傳入request物件，但是會讓app每個view函式多一個引數
    2. 避免view函式有許多不一定用得到的引述，使用context還讓某些物件暫時可被全域操作，寫法如下：
      • 在view函式中，request被當成全域變數使用，但實際上request並非全域變數
      • 在多執行緒伺服器中，各自可以同時處裡來自不同客戶端的不同request，故需要在request內看到不同物件
      • Flask藉由context讓某一執行緒可全域存取某些變數而不干擾其他執行緒

      一個程序通常有多個活動的執行緒，而多執行緒的web伺服器會啟動一執行緒池，並從中選出一個執行緒來處理收到的各個request

  • Flask有兩種context，其公開變數如下表：

    Context 變數名稱 說明
    application context current_app 運行中的app實例
    application context g 在app處理request時可將這物件當成暫存區，該變數會在各request重設
    request context request request物件，封裝用戶端送來的HTTP request內容
    request context session 使用者session(是個字典)，app可用來儲存需要在各request之間記住的值
    • Flask會在發送request給app前，啟動/推動(push)兩個context，並在處裡完request後移除
      • 啟動application context => 可用current_app、g
      • 啟動request context => 可用request、session
      • 若沒有啟動context就存取上述變數則會出錯
      • 在Python殼層session展示application context運作如下 (假設寫app實例的檔名為hello.py)：

        • application context未啟動 => current_app.name錯誤

Request Assignment

  • 當app收到自用戶端的request，必須知道應該呼叫哪個view函式
    • Flask會在app的URL map中尋找request裡面的URL => 存有UAL與其相對view函式的對應關係
    • 建立map會以下列方式取資料：
      • app.route裝飾器
      • app.add_url_rule無裝飾器的版本
      • 可用Python shell查看hello.py的map
  • app中的路由：
    • /、/user/<name>適用app.route裝飾器定義的
    • /static/<filename>是Flask加入的特殊路由 => 用於存取靜態檔案
  • URL map裡的(HEAD, OPTIONS, GET)元素是路由要處理的request方法
    • Flask會自動管理HEAD、OPTIONS方法 => 實際上app的URL map裡的路由都被指派給GET方法

Response

  • 當Flask呼叫view函式時 => 預期回傳值為request的回應，通常是一個被當成HTML網頁回去給用戶端的簡單字串
  • 但是HTTP協定在回應request時需要的是狀態碼
    • 預設是200 => request已被成功執行
    • 若view函式需要不同狀態碼來回應 => 在回應文字後面加數字碼作為第二回傳值
      • 回應也可以使用第三個引數，但要加入HTTP回應標頭字典
  • Flask view函式可以回傳一個回應物件
    • make_response()可接收一到三個引數，讓view函式可回傳相同的值並回傳一等效的回應物件

    • 使用abort()，在URL裡的id動態引數非有效使用者時回傳404
      • abort()沒有將控制權還回函式，因為引發了異常狀況

Web Form

  • 首先會使用Flask-WTF擴充套件
    • 可在處理網頁表單時更輕鬆
    • 此為Flask製做的整合套件，有無關特定框架的WTForms套件
    • 使用pip進行安裝：
  • Flask-WTF
    • 不需要app層級的初始化，但希望app設置密鑰
      • 密鑰：一種隨機且不重複的字串，當成加密/簽署金鑰來使用
      • Flask使用金鑰來防止使用者session的內容被竄改。

      如果有各種不同的app，建議選擇不同的金鑰

      • app.config是儲存Flask、擴充套件、app本身使用的組態變數的通用位置。

    Flask-WTF要求設置密鑰的原因：保護所有表單免於CSRF

    CSRF (Cross-Site Request Forgery，跨站偽造請求)，使用者在無知的情況下點選某外部網站的鏈結(甚至僅是瀏覽頁面)回到已登入網站，即使沒有執行任何JavaScript的情況也能使CSRF攻擊成立，因此又稱 One-click Attack
    從防禦認識CSRF

Form Type

  • 使用Flask-WTF，伺服器裡的每個web表單都是一繼承FlaskForm的類別來表示
    • 此類別定義表單的欄位串 => 每個欄位都是物件
    • 每個欄位物件都可附加一個以上的驗證函式

      可檢查使用者送來的資料是否有效

  • 在app.py中添加下列程式：
    • 將表單欄位定義為類別變數 => 指派與欄位類型有關的物件
      • Nameform表單中有name文字欄位、submit送出按鈕
    • StringField類別：type="text"的HTML<input>元素
      • 選用引數validators定義一串驗證函式 => DataRequired()確保送出的欄位不是空的

      欄位建構式的第一個引數為表單轉為HTML時的標籤

    • SubmitField類別：type="submit"的HTML<input>元素

View Function to Handle the Form

  • 在app.py中，新增一個新的route跟view函式
    • app.route中method引數會讓Flask在URL map裡將view函式註冊為GET、POSTrequest的處理函式
      • 若沒提供 => 註冊為GET處理函式
    • 表單用POST request來處理表單相對方便
      • GET request沒有內文(body)，故會用查詢字串形式將資料附在URL上
    • name：保存表單收到的名字，若無則初始化為None
    • 表單的validate_on_submit()會在表單被送出、所有驗證函式都收到資料 => 回傳True
      • 初次造訪時會收到一沒有表單資料的GET request
        • validate_on_submit()回傳False
        • 跳過if，用轉譯模板來處理request => name設為None後傳入函式
      • 表單由使用者送出，伺服器收到一POST request與資料
      • validate_on_submit()呼叫驗證函式，倘若接受後則回傳true

Redirection and User Session

  • 上面app.py的版本會有一個問題在於：
    • 當輸入名字送出之後，再按重新整理後，可能會看到警告，要求確認之後再次送出表單
    • 原因：重新整理之後會送出上一個request
    • 解決方法：使用轉址來回應POST請求
  • 轉址為一種特殊回應，其內容為URL，而非HTML字串
    • 當瀏覽器收到轉址回應
    • 用GET發出轉址URL

      因為需要送出兩個request，此頁面可能要多花費幾毫秒載入

    • 此法為 Post/Redirect/Get 模式
  • 第二種問題為：
    • 當app處理POST request，可讀取使用者在form.name.data輸入的名字 => 但request結束則資料消失
    • app可將資料放在使用者 session，在兩次request間記得它

      使用者session：

      1. 私用的存放區，每個連接用戶端都可以使用它
      2. 預設情況下會放在用戶端cookie，且會被密鑰加密簽署 => cookie內容更改則簽章失效
      3. 與request context有關變數，稱為session
    • 在app.py中添加下列程式：
    • 前一版的name變數放入session裡面的session['name'] => 可在不同request之間保存
    • 儲存有效表單資料的request會在結束時呼叫redirect() => 產生HTTP轉址回應的Flask協助函式
      • 會接收一轉址URL引數，方法有二：
        1. redirect('/nameform')
        2. redirect(url_for('nameform'))
          • 唯一的引數為端點 (endpoint) 名稱

            端點：每個路由都具備的內部名稱，一般是指view函式

      • render_template()的部分會用session.get('name')直接從session中取得name引數
        • 使用get()查詢，找不到則會回傳None
