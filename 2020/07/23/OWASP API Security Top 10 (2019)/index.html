OWASP API Security Top 10 (2019)

2020-07-23
Cyber Security / API Security
字數統計: 6.9k字   |   閱讀時長≈ 29分鐘
  • REST API 跟 web applications 有相當多的相似點，但根本上還是有差別
  • web application
    • 資料處理會在 server 端；網頁之後轉譯送到 client 瀏覽器
    • 商業網路的端點關聯性少、保護相當簡單（僅設置 WAF）

      WAF = Web-Application Firewall

  • API-based application
    • UI 使用 API 從後端 server 收送資料，提供應用程式的 function
    • client 自行轉意、維護該狀態
  • 本質上，API 和 Personally Identifiable Information (PII) 揭露程式邏輯、機敏資料類似
    • 也因此逐漸成了攻擊者的目標
    • API 保護契約，但不包含足夠的方法確保契約的維護、背後服務的安全性
  • 如此表示需要專屬 API security 的分支來了解、解決 API 獨特的漏洞和資安風險

1. Broken object level authorization

Aspect Description
Attack Vectors 攻擊者替換物件的 ID 來取得其機敏資料
這問題在 API 為背景的應用程式非常常見，因 server 通常會仰賴參數來確定 client 的權限
Security Weakness 授權、存取控制機制在當今應用程式中較複雜且廣泛，開發者可能忘記在存取機敏物件時忘記附上授權確認
Impacts 資料公開給未授權方、資料遺失、資料操控
未授權存取物件可能取得帳戶全部資料

此攻擊在 web security 有另一個名稱 Insecure Direct Object Reference (IDOR)

  • 「Object Level Authorization」是一種存取控制機制

    • 在使用者存取某個其可以存取的物件，code level 被驗證時執行
    • 每個 API 端點
      • 接收物件的 ID
      • 表現物件所有的動作
      • 執行物件階段授權確認 => 應該驗證已登入使用者已存取來對請求物件做請求的動作
    • 機制失敗影響：
      • 未授權資訊公開、修改
      • 所有資料銷毀

  • Scenario

    1. E-commerce
      • 攻擊者觀察到 API 端點以 /shops/{shopName}/revenue_data.json 的格式來取得資料
      • 只要知道商店名稱 => 替換 {shopName} 便能夠取得資料
    2. Monitoring the Network Traffic of a Wearable Device
      • 攻擊者觀察到 HTTP PATCH 的 request 有個 X-User-Id: 54796
      • 將其值改為 X-User-Id: 54795 也能收到成功的 response => 可以修改其他帳戶的帳戶

  • Prevention

    • 根據使用者政策、等第實施適當的授權機制
    • 用授權機制確認已登入使用者存取並表現請求的動作 => 每個 function 會從 client 取得輸入來存取資料庫的紀錄
    • 使用隨機值作為 GUID => 用以紀錄
    • 寫測試來評估授權機制

Reference:

  1. CWE-284: Improper Access Control
  2. CWE-285: Improper Authorization
  3. CWE-639: Authorization Bypass Through User-Controlled Key

2. Broken authentication

Aspect Description
Attack Vectors 1. 在 API 的認證相當複雜且容易困惑 => 軟體與安全工程師經常混淆認證的邊界及實施的正確性
2. 認證對攻擊者來說是很好的目標，因為對所有人開放
Security Weakness 有兩個子議題：
1. 缺乏保護機制 -> 負責認證的 API 端點必須與其他端點不同處置，且需額外保護
2. 機制功能的錯誤執行 -> 執行中的機制並未考慮到攻擊向量或是有錯誤的 use case
Impacts 攻擊者可以控制另一個使用者帳戶，讀取其個人資料、表現機敏行為（e.g. 金錢交易、傳送私人訊息）

  • 認證端點和其流程都應該被保護，且「Forgot password」、「Reset password」兩個功能也應該視同認證機制

  • API 可滲透的機會：

    • 允許 Credential Stuffing Attack => 攻擊者藉此建立有效使用者帳密清單
    • 允許攻擊者在單一使用者帳戶進行 Brute Force Attack => 沒有 captcha 或 account lockout 機制
    • 允許弱密碼
    • 在 URL 中傳送機敏認證資料 (e.g. password, auth token)
    • 未驗證認證 token
    • 接受未簽章或弱簽章的 JWT token (e.g. "alg": "none")，或是未驗證有效期

      JWT = Json Web Tokens

    • 使用明文、未加密、弱 hash 的密碼
    • 使用弱加密金鑰

  • Scenario

    1. Credential Stuffing
      • 利用 lists of know usernames/passwords 進行
      • 應用程式並未實施 Automated Threat / Credential Stuffing 保護 => 應用程式可以用以測試哪組帳密有效
    2. Passwork Recovery Workflow
      • 發布一個 POST request（包含使用者名稱）到 /api/system/verification-codes
      • SMS 的 token (6 bits) 會送到目標手機 => API 並未頻率上限政策，攻擊者可以用多執行緒腳本測試所有可能的組合
      • /api/system/verification-codes/{smstoken} 找到正確 token 僅需幾分鐘

  • Prevention

    • 確認 API 所有可能的驗證流程（手機、網站、實施一鍵認證的深層連結…）
    • 詢問工程師可能缺漏的流程
    • 詳讀認證機制

      OAuth 既非認證，也非 API 金鑰

    • 不在認證、token 產生、密碼儲存創造 wheel，使用標準
    • 身份驗證恢復/忘記密碼端點必須與登入端點同等對待 => brute force / rate limiting / lockout 的保護
    • 使用 OWASP Authentication Cheatsheet
    • 執行多重認證
    • 在認證端點執行反暴力破解機制來緩解 Credential Stuffing, Dictionary Attack, Brute Force Attack => 機制應該相較嚴苛
    • 執行 Account Lockout / Captcha 機制 => 預防 Brute Force、執行弱密碼偵測
    • API 金鑰不應該使用使用者認證，但使用 Client App/Project Authentication

References:

  2. Authentication Cheat Sheet
  3. Credential stuffing
  4. CWE-798: Use of Hard-coded Credentials

3. Excessive data exposure

Aspect Description
Attack Vectors 大都透過分析 API 回應來對流量進行 sniffing
可以找到暴露的機敏資料
Security Weakness API 根據 client 來進行資料過濾 => API 通常作為資料來源，有時候開發不會思考到機敏資料暴露的可能
自動化工具難以偵測 => 不好區分 API 回傳的資料是合理的還是機敏資料
Impacts 機敏資料暴露

  • 在設計上就有回傳機敏資料的缺失

    • 通常會被 client 端過濾掉
    • 攻擊者可以簡單以 sniffing 來查看機敏資料

  • Scenario

    1. Mobile Team
      • 使用 /api/articles/{articleId}/comments/{commentId} 端點 => 在 article view 轉換 comment metadata
      • Sniffing 手機應用程式的流量，攻擊者可以找到其他機敏資料（與 comment 作者有關）被一併回傳
      • 端點執行通用 toJSON() 在 User 的 model => 其中包含 PII 能序列化物件
    2. IoT-based Surveillance System
      • 允許管理者創立不同權限的使用者
      • 一個管理者創立使用者帳戶作為新的資安管理員 => 僅能存取特定網站架構
        • 如果使用其手機應用程式 => API 會觸發 /api/sites/111/cameras 來接收有效監視器檔案並顯示在 dashboard
      • response 包含格式：{"id":"xxx", "live_access_token": "xxxx-bbbbb", "building_id":"yyy"}
        • 但 client GUI 僅顯現哪個資安管理員存取

  • Prevention

    • 不依賴 client 端過濾機敏資料
    • 在 API 看 response 來確認其中不包含機敏資料
    • 後端工程師在釋出新的 API 端點應理解「資料的目標客戶」
    • 避免使用通用方法 (to_json(), to_string()) => 挑選特定屬性
    • 將機敏資料跟 PII (Peersonally Identifiable Information) 分類讓應用程式儲存和使用，回顧所有 API 呼叫 return 來確定 response 不含資安議題
    • 執行 schema-based response 驗證機制 => 其中定義並執行所有 API method 的資料回傳（包括錯誤訊息）

References:

  1. CWE-213: Exposure of Sensitive Information Due to Incompatible Policies
    [color=#E6CAFF]

4. Lack of resources and rate limiting

Aspect Description
Attack Vectors 需要 API request；無需認證
同時的 request 便可以單一本機電腦或雲端運算資源呈現
Security Weakness 在 API 很常看到未對執行效能設限，或沒有妥當設置
Impacts 可能導致 DoS => API 無反應甚至無法運作

  • API 的 request 會消耗資源（網路、CPU、記憶體、儲存單位）=> 其數量根據使用者輸入、端點商業邏輯來決定

    • 還需考慮到多個 API client 的 request 會搶資源

  • API 可滲透的機會：（至少其中一點沒有設置或設置不妥）

    • Execution timeouts
    • Max allocable memory
    • Number of file descriptors
    • Number of processes
    • Request payload size
    • Number of request per client/resource
    • Number of records per page to return in a single request response

  • Scenario

    1. Large Image
      • 攻擊者上傳大圖片 => 用 POST 送 request 到 /api/v1/images
      • 當上傳完成，API 也建立複數個不同尺寸的縮圖 => 記憶體會為了這些縮圖而造成 API 無法回應其他請求
    2. User List
      • 在 UI 包含使用者清單（限 200 筆每頁）=> 使用者清單透過 /api/users?page=1&size=100 從 server 取回
      • 攻擊者改變 size 參數到 200,000 => API 無法回應，同時無法處理其他 client 更多 request (DoS)

  • Prevention

    • 用 Docker 對 memory, CPU, number of restarts, file descriptor, processes 設限
    • 在定義的時間範圍內執行 client 能呼叫 API 的次數限制
    • 告知 client 超過限制次數和時間，限制會重置
    • 新增合適的 server 端對 query 和 request body 參數的驗證，特別是控制 response 中紀錄回傳數量
    • 定義並強制資料的最大尺寸（字串最大長度、Array 元素最大數量） => 對所有輸入的參數、payload

References:

  1. Blocking Brute Force Attacks
  2. Docker Security Cheat Sheet - Limit resources (memory, CPU, file descriptors, processes, restarts)
  3. REST Assessment Cheat Sheet
  4. CWE-307: Improper Restriction of Excessive Authentication Attempts
  5. CWE-770: Allocation of Resources Without Limits or Throttling
  6. Security Strategies for Microservices-based Application Systems

5. Broken function level authorization

Aspect Description
Attack Vectors 攻擊者送正常 API 呼叫到 API 端點，但其不應該被存取 => 可能會暴露匿名使用者或無特權使用者
自從 API 結構化後很容易發現這個漏洞 => 存取的方法更容易被預測（e.g. 將 HTTP 的方法從 GET 改為 PUT、將 URL 中 users 改為 admins）
Security Weakness 確認功能或來源的授權通常由 config 管理，有時候是 code level
現代的應用程式包含許多角色、群組、複雜的使用者等級 => 很難確認
Impacts 允許攻擊者存取未授權功能
管理功能會是這類攻擊的關鍵目標

  • 找到這類議題最好的方法：深入分析授權機制，注意使用者等級制度、不同角色或群組，思考幾個問題：

    • 常規使用者能存取管理端點否？
    • 使用者能透過改變 HTTP 方法 (e.g. GET -> DELETE) 來進行機敏動作 (e.g. creation, modification, erasure) 否？
    • 使用者可以透過改變端點 URL 和參數 (e.g. /api/v1/users/export_all) 從群組 X 存取到群組 Y 才能存取的功能否？

  • 別從 URL 假設該 API 端點是常規的還是管理的

  • 開發者可能選擇在特定關聯的路徑暴露更多的管理端點 (e.g. api/admins) => 相當常見用來找管理端點

    • 其他像是 api/users 也是類似的意思

  • Scenario

    1. Invitation
      • 在一個應用程式的註冊階段僅允許受邀使用者的加入
      • 手機應用程式觸發 API 呼叫 GET /api/invites/{invite_guid} => response 包含 JSON，其中有邀請的細節（使用者角色、使用者 E-mail）
      • 攻擊者複製 request 並修改 HTTP 的方法和端點 POST /api/invites/new => 此端點僅能由管理員透過管理終端存取，但卻沒有執行 function level authorization check
      • 攻擊者可以送管理帳號邀請給自己POST /api/inites/new {"email":"hugo@malicious.com", "role":"admin"}
    2. Administratiors’ Endpoint
      • API 包含僅有管理員看得到的端點 GET /api/admin/v1/users/all => 會回傳所有應用程式使用者的資料、未執行 function-level authorization check
      • 攻擊者理解 API 架構後對端點進行猜測與管理存取 => 取得應用程式使用者機敏資料

  • Prevention

    • 應用程式保持一貫性，授權模組相對簡單 => 引用其他商業功能
    • 保護機制應該提供一個以上應用程式的外部元件
      • 執行機制預設拒絕所有存取，需要特定角色明確授權來存取每個功能
      • 回顧 API 端點比照 function level authorization 漏洞，尤其應用程式和群組等級的商業邏輯
      • 確定所有管理控制器繼承管理員概念控制器 => 執行使用者群組或角色的授權確認
      • 確定在常規控制器的管理功能執行使用者群組或決色的授權確認

References:

  1. Forced browsing
  2. OWASP Top 10 2013-A7-Missing Function Level Access Control
  3. OWASP Development Guide: Authorization
  4. CWE-285: Improper Authorization

6. Mass assignment

Aspect Description
Attack Vectors 需要了解商業邏輯、物件關聯、API 架構
在 API 中較容易做到 => 從設計上揭露潛在的應用程式執行的屬性名稱
Security Weakness 當今的框架鼓勵開發者使用功能 => 自動從 client 的變數、內部物件輸入
攻擊者可以利用這個方法來 update 或 overwrite 機敏物件屬性
Impacts 提權 (privilege escalation)、資料損害 (data tampering)、安全機制繞過 (bypass of security mechanism)等等

  • 在現代應用程式的物件可能包含許多屬性

    • 有些可以直接從 client 進行 update (e.g. user.first_name、user.address)
    • 有些不應該如此 (e.g. user.is_vip)

  • API 的端點若自動改變 client 內部屬性參數（未考慮到機敏性與揭露的層級）=> 攻擊者可以更新物件屬性，比如幾個機敏屬性：

    • Permission-related properties
      • user.is_admin、user.is_vip
      • 應該僅由 admin 設置
    • Process-dependent properties
      • user.cash
      • 應該僅在交易認證後內部設置
    • internal properties
      • article.created_time
      • 應該僅由應用程式內部設置

  • Scenario

    1. Edit Profile
      • 在共乘應用程式中提供使用者編輯個人基本資料
      • 在此程序中，呼叫 API => PUT /api/v1/users/me 加上接續的 JSON 物件 {"user_name":"inons", "age":24}
      • request 包含額外的 credit_balance 屬性 => GET /api/v1/users/me、{"user_name": "inons", "age":24, "credit_balance":10}
      • 攻擊者重送 request，並改成 payload => {"user_name":"attacker", "age":60, "credit_balance":99999}
    2. New Property
      • 影音共享網頁允許使用者以不同格式上傳、下載
      • 攻擊者發現在 API 端點 GET /api/v1/videos/{video_id}/meta_data 會回傳 JSON 物件（含影片屬性）
        • 其中一個屬性 "mp4_conversion_params":"-v codec h264" => 標明應用程式使用 shell command 來轉換影片
      • 攻擊者同樣發現 POST /api/v1/videos/new 可以 mass assignment 滲透、允許 client 設置任何屬性到影片物件
        • 攻擊者設置 "mp4_conversion_params":"-v codec h264 && format C:/" => 只要攻擊者下載影片就會觸發 shell command injection

  • Prevention

    • 避免自動 bind client 的輸入到程式變數或內部物件
    • 設置屬性白名單 => client 可以更新
    • 使用內建屬性黑名單 => client 不可存取
    • 針對輸入資料的 payload 明確定義並執行 schema

References:

  1. CWE-915: Improperly Controlled Modification of Dynamically-Determined Object Attributes
    [color=#E6CAFF]

7. Security misconfiguration

Aspect Description
Attack Vectors 攻擊者會嘗試找到未修補的漏洞、常見的端點、未保護的檔案和指標 => 增加授權或去熟悉該系統
Security Weakness 發生在 API stack 的任何階層
自動化工具能夠偵測、滲透這些非必要服務或資產
Impacts 能造成機敏使用者資料暴露、系統資訊外洩導致伺服器損害

  • 幾個滲透的可能：

    • 合適的安全強化在應用程式任何一個部分遺失，或在雲端服務有不合適的權限設置
    • 未安裝最近的安全修補程式，或系統過於陳舊
    • 開了非必要的功能 (e.g. HTTP verbs)
    • 未使用 Transport Layer Security (TLS)
    • 安全指令未送到 client (e.g. Security Headers)
    • 未使用 Cross-Origin Resource Sharing (CORS) 或設置不當
    • 錯誤訊息包含 stack 痕跡，或其他機敏資訊暴露

  • Scenario

    1. File with Command

      • 攻擊者在伺服器 root 目錄中找到 .bash_history 檔案 => 包含 DevOps 團隊存取 API 時的指令
      • 同時也找到 API 上僅被 DevOpts 團隊使用的新的節點，且不在文件上

    2. Default Configuration

      • 針對特定的服務，攻擊者利用搜尋引擎找能夠直接透過網路存取的電腦
      • 攻擊者找正在跑主流資料庫管理系統的機器、監聽其預設埠號
      • 主機使用預設配置（預設關閉認證） => 攻擊者新增存取 PII、個人偏好、認證資料
      • 攻擊者在檢視手機應用程式的流量時發現不是所有 HTTP 流量都有安全協定 (e.g. TLS)
        • 尤其在下載圖片檔時
      • 使用者的互動是以二進位 => 雖然 API 的流量是走安全協定，攻擊者找到 API response 的模式
        • 可透過轉譯的內容 (e.g. profile images) 用來追蹤使用者習慣

  • Prevention

    • API 生命週期 (API life cycle) 應包含：
      • 重複的強化來更快、更簡單的部署
      • 回顧、更新所有 API stack 的配置，回顧應包括：
        • orchestration files
        • API components
        • cloud services (e.g. S3 bucket permission)
      • 給所有 API 與靜態資產互動的安全通訊通道
      • 自動的程序來持續地評估所有環境的配置與設定有效性
    • 更多：
      • 預防從攻擊者送的 exception 追蹤和其他有漏洞資訊，若適用則定義並執行在所有 API response payload schemas（包括錯誤回應）
      • 確定 API 僅能被特定的 HTTP verbs 存取，其他則不行 (e.g. HEAD)
      • API 預期從 browser-based client 存取 (e.g. WebApp front-end) => 執行合適的 CORS 政策

  4. CWE CATEGORY: Configuration
  5. CWE CATEGORY: 7PK - Errors

8. Injection

Aspect Description
Attack Vectors 攻擊者會透過各種可能的注入方式餵有害資料給 API => 預期讓它轉譯出來
Security Weakness 注入漏洞非常常見 => 大都在
1. SQL query
2. LDAP query
3. NoSQL query
4. OS commands
5. XML parser
6. ORM
可以使用 scanner 或 fuzzer 找到注入點
Impacts 可能會導致資訊公開、資料損失、DoS、取得主機掌控

  • 滲透的可能性：

    • Client 提供的資料未認證、過濾、淨化
    • Client 提供的資料直接使用或間接接到 SQL / NoSQL / LDAP query、OS command、XML parser、Object Relational Mapping (ORM) / Object Document Mapping (ODM)
    • 來自外部系統的資料未認證、過濾、淨化

  • Scenario

    1. Vulnerable Firmware
      • 源控制裝置的韌體提供端點 /api/CONFIG/restore => 將 appId 分成很多部分的參數送出去
      • 使用 decompliler => 攻擊者發現 appId 直接送到系統呼叫（沒有任何過濾）
      • 改為下列 payload 便可以關掉任何有相同漏洞韌體的設備
    2. Remove booking
      • 有個基礎 CRUD 功能運算的訂房系統應用程式
      • 攻擊者發現 NoSQL 注入可能可以透過 bookingId 來刪掉訂房請求 => DELETE /api/bookings?bookingId=678
      • API 伺服器用下列功能來處理刪除請求
      • 攻擊者攔截 request 並更改 bookingId 的參數 => 能刪除其他使用者的訂房

  • Prevention

    • 資料認證使用單一、可信任、仍在維護的 library
    • 認證、過濾、淨化所有 client 提供的資料或其他外部系統資料
    • 特數字元應該避免使用
    • 偏好安全 API => 提供參數化介面
    • 限制回傳資料的數量 => 避免在 injection 中引發 mass disclosure
    • 認證輸入的資料 => 足夠的過濾器來取每個輸入參數中有效的值
    • 定義資料類型、限制字串參數的方式

References:

  1. Injection Flaws
  2. SQL Injection
  3. NoSQL Injection Fun with objects and Arrays
  4. Command Injection
  5. CWE-77: Improper Neutralization of Special Elements used in a Command (‘Command Injection’)
  6. CWE-89: Improper Neutralization of Special Elements used in an SQL Command (‘SQL Injection’)

9. Improper assets management

Aspect Description
Attack Vectors 舊的 API 版本經常修正 => 容易對沒有更新安全機制的系統造成損害
其更新過的安全機制可能保護最近的 API 版本
Security Weakness 舊的文件使其難以找到和/或修補漏洞
沒有資產清單、不在使用的策略 => 還在運作的未修補系統，進而造成機敏資料洩漏
為了方便部署和獨立系統 => 常找到非必要公開的 API host
Impacts 增加機敏資料存取、透過過時而為修補的 API 版本（連到相同資料庫）取得系統掌控

  • API 被滲透的可能：

    • API host 不乾淨原因，表示有未能明確回答的幾個問題：
      • 哪個環境的 API 正在跑 (e.g. production, staging, test, development)？
      • 誰能存取 API 的網路 (e.g. public, internal, partners)？
      • 哪個 API 版本在運行？
      • 哪些資料被 API 搜集並處理 (e.g. PII)？
      • 資料的流程是什麼？
    • 沒有文件 / 存在的文件未更新
    • 沒有針對 API 版本汰換的企劃
    • 主機的清單消失或過舊
    • 合併的服務清單 (first- / third-pary) 消失或過舊
    • 舊的 API 版本仍在運行，但並未修補

  • Scenario

    1. Old Version API
      • 在重設計應用程式後，本地搜尋服務留下了一個舊的 API 版本 => api.someservice.com/v1 還在運行（能存取使用者資料庫）
      • 針對其中一個最近的應用程式，攻擊者發現其位址為 api.someservice.com/v2 => 將 v2 替換成 v1 存取到舊的、未保護的 API
        • Personal Indentifiable Information (PII) 的洩漏
    2. Beta Version API
      • 社交網路執行 rate-limiting 機制 => 阻擋攻擊者使用 brute-force 來猜測重設密碼 token
        • 此機制不會作為 API 一部分執行，但作為獨立的部分（在 client 跟 official API 間）=> www.socialnetwork.com
      • 有找到 beta API 主機 => www.mbasic.beta.socialnetwork.com
        • 跑在相同的 API，包含重設密碼機制，但沒有 rate-limiting 機制
      • 如此可以重設任何使用者的密碼 => 利用 brute-force 來破 6-bit token

  • Prevention

    • 列所有 API 主幾清單，紀錄每個 API 重要部分，尤其
      • API 環境 (e.g. production, staging, test, development)
      • 誰能存取 (e.g. public, internal, partners)
      • API 版本
    • 列融合的服務，紀錄其在系統中的角色、資料如何交換、其機敏性
    • 紀錄所有 API 各部分的
      • authentication
      • error
      • redirect
      • rate limiting
      • cross-origin resource share (CORS) policy
      • endpoint
        包括 parameter, request, response
    • 自動產生文件 => 把文件建在 CI / CD pineline
    • 製作 API 文件並授權給各個 API 使用
    • 使用外部保護方法 => API security firewall 保護所有公開的 API
    • 避免透過非產品的 API 來使用攢品資料，若無法避免則端點應該與非公開端點有相同的保護機制
    • 新版本 API（包含安全更新）出來後，應該對舊版本做威脅分析

References:

  1. CWE-1059: Incomplete Documentation
  2. OpenAPI Initiative

10. Insufficient logging and monitoring

Aspect Description
Attack Vectors 攻擊者在缺乏紀錄、監控的系統能不被發現的濫用系統
Security Weakness 沒有或不充足的紀錄、監控 => 基本上不可能追溯可疑的活動並即時做出反應
Impacts 再看不到有害活動的狀況下，攻擊者有大把時間損害系統

  • API 滲透的可能：

    • 並未產生任何的 log / 並未正確設置 log 的層級 / log 訊息並未包含細部內容
    • log 完整性無法保證
    • log 並非連續監控
    • API 的基礎架構並沒有連續監控

  • Scenario

    1. Public Access Key
      • 管理性 API 的存取金鑰被放到公開的 repository
        • 其擁有者被 email 提醒，但並未在 48 小時內應對
      • 存取金鑰可能被用來存取機敏資料 => 因不充分的紀錄，無法評估什麼資料被存取
    2. Credential Stuffing Attack
      • 影音分享平台被大規模 credential stuffing attack
      • 儘管 login 失敗會被紀錄，但在攻擊時沒有觸發告警 => 直到使用者反應才會分析 API 的 log
      • 公司發公告請使用者重設密碼，並回報管理部門授權

  • Prevention

    • 紀錄所有：
      • failed authentication attempts
      • denied access
      • input validation error
    • 應該使用適合紀錄管理方案的格式來進行紀錄，應該包含能辨識有害使用者的細節
    • 紀錄應能掌握機敏資料，其完整性應該受保證
    • 配置監控系統來持續的監控整個架構、網路、API 功能
    • 利用 Security Information and Event Management (SIEM) 系統來統合管理所有 API stack 跟 host 的紀錄
    • 配置客製的 dashboard 與 alert => 能更早偵測並回應可疑活動

  2. OWASP Proactive Controls
  3. OWASP Application Security Verification Standard V7
  4. CWE-223: Omission of Security-relevant Information
  5. CWE-778: Insufficient Logging
