Next-Generation Fuzzing

2020-01-31
Cyber Security / Gray Hat Hacking
字數統計: 4.8k字   |   閱讀時長≈ 22分鐘
  • fuzzing，或稱fuzz testing，中譯為模糊測試，為一種滲透測試（軟體測試、漏洞挖掘）的技巧。
  • fuzzing今昔比較：
    • 原本是用隨機產生的資料來對黑箱的軟體或硬體進行測試
    • 如今的工具已經支援黑箱與白箱的測試方法，並且有許多可調整的參數
  • 測試參數功能：
    • 直接性影響fuzzing的過程
    • 針對特定問題對測試過程進行微調

Introduction to Fuzzing

  • 傳統的黑箱軟體測試
    • 從漏洞研究的觀點來看非常的有趣，因其完全不需要了解軟體的內部機制
    • 只需知道哪些介面能夠操作軟體、產生的資料從哪些介面送入
  • fuzzing / fuzz testing
    • 傳統：
      • 用隨機產生的資料測試某類型軟體或硬體
      • 產生的資料有個問題 => 太過簡化
      • 有時無需任何針對軟體運行機制或輸入資料架構的背景知識
    • 如今：
      • 不再引用隨機產生的資料
      • 使用不同方法來檢驗

Types of Fuzzers

  • fuzzing 其實並非一個精確的科學方法 => 可以多多測試不同的fuzzing跟其參數👍
  • 基於資料產生的演算法，fuzzing大致可分為：
    • Mutation fuzzer
    • Generation fuzzer
    • Genetic or evolutionary fuzzer

Mutation Fuzzers

  • Mutation-based fuzzer，或稱dumb fuzzer
  • 最簡單的變種，最接近原本亂數資料輸入的概念
    • 轉換輸入資料，通常亂數產生
    • 接著將資料送到目標軟體來試圖引發crash

Generation Fuzzers

  • 又稱grammar-based fuzzer，或是white box fuzz testing
    • 需要對內部運作的協定有先備知識
  • 基於假設有效率的測試需要對目標機器內部運作有一定了解
    • 無需有效資料輸入的範例、協定表現
    • 能產生基於描述資料或協定結構之data model的測試例
    • 這些model通常寫成config檔 => 格式依不同工具而有所差別
  • 主要問題：寫data model
    • 若有些簡單的協定或資料結構有文件的話就不會有這個問題
    • 但！是！這！樣！的！案！例！非！常！罕！見！
    • 現實案例複雜的多
      • 規格、文件的可得性還得經過相當的努力來轉換成fuzzing model
      • 要是公司沒有按照規格…或是有些微調整…甚至是規格書上沒說的新功能…

Genetic Fuzzing

  • 也稱為evolutaionary fuzzing => 以工具決定最好的輸入測試資料
    • 基於程式測試能包含處理的最大範圍
  • 大多測試都會在變動輸入時弄個告示
    • 在接觸到新的源碼區塊時
    • 儲存變動過的測試輸入
  • 這樣fuzzing工具能夠找到「最佳存活」方法

Mutation Fuzzing with Peach

所有操作會在Lab統合操作

所需工具：
Peach Community Edition
Mono

  • Peach framework => 跨Windows、Linux、MacOS X

    Linux、MacOS X => 一個跨平台的的 .NET 開發框架 Mono 必須要run
    如果不同平台的話，操作步驟可能會有些許差異

  • Mutation fuzzing從使用者角度來說不太需要太多作業 => 選一組sample後送進變異程式便能開始fuzzing
  • 開始利用Peach來fuzzing
    • 先創建Pit檔案 => XML文件，其中包括所有fuzzing session設定，幾個常見資訊：
      • General configuration
        • 定義與fuzzing無關參數
      • Data model
        • 定義資料結構 => 會被測試成Peach的特殊語法
      • State model
        • 定義狀態機必須確認當前協定 => data model不足以抓到所有協定
      • Agents and monitors
        • 定義方法：
          • 分散fuzzing工作量
          • 監控目標軟體: 標示錯誤或漏洞
      • Test configuration
        • 定義讓Peach建立各個測試例，fuzzing策略能修改資料
  • Mutation Pit
    • 相當容易建立
    • Peach提供不同模板因應不同場景來執行與修改
    • 設定檔可創建並用任何文字編輯器修改
  • rm_fuzz.xml
  • Pit檔包括許多重要的部分會影響並決定fuzzing的過程，下面說明這些部分怎麼影響fuzzing過程：（以上面程式碼為例）
    • DataModel
      • 定義會被fuzzing的資料結構

        黑箱測試時通常都不知道

      • 會額外使用單一資料項來表示<Blob />
        • 描述隨機二進位資料組
        • 不會執行任何資料限制
      • 要是忽略資料結構
        • 不會定資料型別及個別的大小 => 不精確的方法
        • 會縮短fuzzing的時間
      • 黑箱fuzzing非常快、且設定的價格低廉
        => 通常值得用好的資料結構
    • StateModel
      • 定義在進行fuzzing中資料的不同狀態
        • 非常簡單 => 因為僅有單一檔案、僅為了測試
      • Fuzzing網路協定用來解釋state model
        • 為在實現協定時運行不同狀態 => 必須正確走過狀態圖
      • 定義StateModel的功用：
        • 指示fuzzer走狀態圖
        • 能測試更多的原始碼跟功能
    • Agent
      • 定義debugger => 監控執行的目標程式、蒐集crash的資訊
      • crash的資料之後手動檢視、分類(分成有關或無關)
        • 有關的crash需重新檢查可突破狀況來給值
    • Test
      • 定義與fuzzing有關的狀態選項
        • 以此例來說 => 定義filename來產生測試例fuzzed.rm -> 定義logs當作程式crash資料的紀錄導引
  • Peach有許多方法測試Pit的正確性
    • 加上--test參數 => 會先parse整個Pit後回報錯誤
    • 開始一個新的Peach session
    • Ctrl + C停下fuzzer => 確定機器運行時的維護
      • 終止session的結果會放在log目錄中
        • 命名法則 => [檔名]_[時間].txt
        • 包括session資訊(測試數量、crash資訊)
      • 倘若session成功運行會多一個Faults目錄
        • 包含針對各類crash分門別類的目錄，分別有一個以上的測試例
        • 包括下面資訊：
          • 引發crash的變異測試例
          • 引發crash時程式階段的debug report => 有暫存器的狀態和值、部分的stack內容
          • 原本的測試例
  • session可以跳過已經測試過的測試例而直接回復
  • 其他方法去看fuzzer進展 => 在command-line的輸出，會放在一連串反覆數字的第一個
    • e.g. 第十三個測試例
  • --skipto可以跳到特定的測試例

Generation Fuzzing with Peach

所有操作會在Lab統合操作

  • 接著預期發現Stephen Bradshaw’s Vulnserver的漏洞

    Vulnserver是一個為了學習fuzzing和exploit的server
    stephenbradshaw/vulnserver
    其中作者也有專門的部落格來說明：The Grey Corner

    • 為預編譯的binary檔 => 在Windows上啟動
    • 注意：這個server要跑在獨立的測試系統、或是host-only的虛擬機上
    • 可以透過netcat來連到server
    • VulnServer有效的命令參數：
      • HELP
      • STATS [stat_value]
      • RTIME [rtime_value]
      • LTIME [ltime_value]
      • SRUN [srun_value]
      • TRUN [trun_value]
      • GMON [gmon_value]
      • GDOG [gdog_value]
      • KSTET [kstet_value]
      • GTER [gter_value]
      • HTER [hter_value]
      • LTER [lter_value]
      • KSTAN [kstan_value]
      • EXIT
  • 接著就是寫Peach Pit => 主要改的部分是DataModel的部分
    • 顯示有效應用程式指令的架構
  • 為了簡化測試，僅測試TRUN指令
    • 原因居然是因為這個指令聽起來很帥
    • 這個Peach Pit是基於David Um的範例，其本身則是基於更早由Dejan Lukan的文章
  • TRUN_fuzz.xml
  • 在這個generation-based的Peach Pit和前一個Mutation Peach Pit有下列主要的不同：
    • DataModel
      • 改為對TRUN指令語法的描述 => TRUN \r\n
      • 其中空白的部分為可變的 (可被fuzzing)
    • Agent
      • 改為顯示遠端Peach代理監控應用程式的進程
        • 如果有必要的話會重啟
    • Publisher
      • 改為顯示Peach的TCP連線能力
      • 給漏洞應用程式的位址、埠號
  • 為了跑這個Peach Pit => 啟動Peach agent
    • 跑Peach Pit
    • 可能會出現以下情形
      • fuzzer會找到exception並重做一次

Crash Analysis

  • 在fuzzing過程要是都順利的話，理應出現目標應用程式crash的log
  • 幾個常見crash的痕跡：
    • 紀錄簡單檔案或資料 => 先做紀錄，能夠用以重現crash
      • file fuzzer ~ 一個簡單的測試檔案會被儲存、標示，用以做回顧
      • network application fuzzer ~ 一個PCAP檔可能會在應用程式偵測到crash時被紀錄、儲存
      • 紀錄簡單檔案或資料是最基本追蹤應用程式crash的方法
      • 未提供crash的上下文
    • 應用程式crash的log檔可以透過各種方法來蒐集
      • 一般debugger就是用以監控目標應用程式狀態、偵測任何crash的痕跡
      • 若偵測到crash -> 會蒐集CPU環境資訊(e.g. 暫存器狀態、記憶體堆疊) -> 存成獨立crash sample檔
      • crash log在理解crash類型上很好用
        • 在沒有環境資訊的狀態下很難判斷有幾種不同的漏洞 => 因為一個程式可能因為同一個bug報錯數百遍
        • log給的就是初步去篩選、分類出不同的漏洞
    • 偵測到crash時，有很多客製化的腳本(蒐集了許特定種類的資訊)可以運行
      • 最簡單：運行debugger的套件
        • e.g. !Exploitable是一個挺好用的debugger套件(for Microsoft WinDbg)，用來確認一個crash可exploit否
        • 雖然套件好用，能給crash資訊並分類，但不能盡信
      • 要全面性的確認crash能不能exploit => 手動分析
        • 因為有時候仰賴研究員來決定漏洞的價值
  • Peach用WinDbg、!Exploitable => 蒐集具上下文的crash資訊、顯示crash串列
    • 會將所有crash資訊放在Fault目錄下
    • 進一步到測試目錄中，會發現幾個資料夾列在Fault目錄中
    • 再更進一步會發現到真實測試例ID以及其內容

      範例ID為185

      • 其中RemoteAgent.Monitor.WindowsDebugEngine.description.txt包括crash最好的資訊，其中有兩個主要部分：
        • debugger蒐集的crash資訊：
          • 讀取模組名稱
          • CPU暫存器資訊
          • 記憶體片段
        • !Exploitable回報的crash資訊、分類
          • 有更多crash的上下文：
            • exception code
            • stack frames資訊
            • bug title
            • 分類
              • !Exploitable推斷crash潛在的可突破性
              • 四種可能的值：Exploitable, Probably Exploitable, Probably Not Exploitable, Unknown

Genetic or Evolutionary Fuzzing with AFL

  • genetic / evolutionary fuzzing最好的公具 => AFL
    • 一般都用C / C++寫的(for file-based parser)
  • 當一個應用程式充滿了if/else，通常很難(假設是有可能的)用mutation fuzzer去跑，恐怕需要一輩子的時間才跑得完
    • e.g.
      • file.txt
      • asdf3.c
        • abort()會造成程式crash，但問題就在於fuzzer有沒有辦法找到這個錯誤
      • 用mutation fuzzer => 送一個檔案，有1 / 2568的機會能觸到最中心的程式碼
        • 假設電腦每秒處理1000個檔案，跑完所有可能性的上限 = (256**8/1000/86400/365) 584942417L
  • 下面用AFL來處理這個問題
    • 先用AFL工具編譯
    • 接著開始fuzzing
    • 可以看到AFL出現一個information-packed的介面
      • 最後fuzzer會找到一個crash (聽說大概會花五分多鐘的話，那麼可以確定那台電腦很貴)
      • AFL也提供crash log
        • 如同預期的，前八碼abcdefgh觸及核心源碼塊，觸發程式abort

LAB

  • 按照下列步驟完成Lab：
    1. 下載Kali Linux (2GB RAM、雙核心)並裝在虛擬機上

      MacOS可直接用brew install afl-fuzz安裝

    3. 複製完整的asdf3.c檔案

    8. 檢視前述crash的GUI、crash log
